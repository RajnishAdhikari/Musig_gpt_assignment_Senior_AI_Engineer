# MusicGPT Search Integrity Challenge Decisions

## Part 1 - Schema Design

**Short summary:** The dataset is a DynamoDB-style export, so ingest first normalizes wrapped values into standard Python/Postgres types. I promote fields that are frequently searched, filtered, or ranked, such as `title`, `prompt`, `bpm`, `musical_key`, `primary_genre`, `primary_mood`, vocal attributes, tags, and `acoustic_prompt_descriptive`. I keep the full unwrapped metadata in JSONB so the schema remains maintainable if new metadata fields appear later. I model `conversion_path_1` and `conversion_path_2` as child variations under the same generation because they share sonic lineage and should not be treated as unrelated songs. For embeddings, I use `acoustic_prompt_descriptive` when available and fall back to `sounds`, then prompt/title/tags when it is missing.

**Simple example:** A raw export field like this:

```json
{
  "technical": {
    "M": {
      "bpm": { "N": "110" },
      "key": { "S": "C major" }
    }
  },
  "all_tags": {
    "L": [
      { "S": "pop" },
      { "S": "happy" }
    ]
  }
}
```

is ingested as normal values:

```json
{
  "technical": {
    "bpm": 110,
    "key": "C major"
  },
  "all_tags": ["pop", "happy"]
}
```

Then `bpm = 110`, `musical_key = "C major"`, and `all_tags = ["pop", "happy"]` are searchable columns on each output row, while the complete unwrapped metadata is still stored in `metadata` JSONB.

**Decision made:** I model the export record as a `generations` parent row and each `conversion_path_1` / `conversion_path_2` value as a separate `songs` row. The searchable output has its own id, conversion path, lyrics, sounds, engagement counters, embedding, and generated `search_vector`, while both siblings share `lineage_id`.

**Why:** Users play and click an output, not a parent generation, so engagement belongs on `songs`. Diversity needs the shared parent lineage, so keeping `lineage_id` explicit avoids treating sibling conversions as independent songs.

**DynamoDB wrappers:** `seed.py` recursively unwraps `{ "S": ... }`, `{ "N": ... }`, `{ "L": ... }`, `{ "M": ... }`, `{ "BOOL": ... }`, `{ "NULL": true }`, `SS`, and `NS`. Unknown or sparse fields are retained in `metadata` JSONB instead of dropped.

**First-class columns:** I promote fields that are common search/filter/ranking features: `title`, `prompt`, `sounds`, `acoustic_prompt_descriptive`, `bpm`, `musical_key`, `duration_type`, `primary_genre`, `primary_mood`, `vocal_genders`, and `all_tags`. I keep the rest in JSONB because the export shape can evolve without a migration for every new tag family.

**Embedding text fallback:** `acoustic_prompt_descriptive` is the embedding source because it is the richest compact sonic description. If it is missing, seed falls back to `sounds`, then prompt/title/tags. The fallback is implemented in `choose_embedding_text()` so a sparse record still gets a non-null embedding text.

**Embedding implementation:** `search.py` uses `sentence-transformers/all-MiniLM-L6-v2` to embed the selected `embedding_text` into a 384-dimensional vector. This is a text semantic model, not a music-audio model, and that is intentional for this assessment. The searchable evidence supplied to the system is text metadata: title, prompt, sound descriptions, tags, key, BPM, vocals, and the unwrapped acoustic prompt. The user query is also text. MiniLM is therefore aligned with the actual retrieval input, gives a real semantic baseline for phrases like "chill study beat" or "bright holiday pop", runs on CPU, requires no paid external service key, and is small enough for reviewers to run locally. Exact technical terms such as `C major`, `128 BPM`, and `female vocal` are still protected by the weighted Postgres full-text channel rather than delegated only to embeddings.

**Embedding migration:** `models.py` sets `VECTOR_DIMENSIONS = 384`, and Alembic migration `20260502_0002_minilm_embedding_dimensions.py` changes `songs.embedding` to `vector(384)`. Existing vectors are cleared during the migration and regenerated by `seed.py`, because old vectors from a different embedding method are not comparable to MiniLM vectors.

**Simple embedding example:** For a record whose `acoustic_prompt_descriptive` says "A bright pop holiday track at 110 BPM in C major...", the vector is built from that descriptive sentence. A query like "festive upbeat pop" can match semantically through MiniLM even if the exact phrase does not appear, while a query like "C major female vocal" is rescued by the full-text rank because those are precise metadata terms.

**Why not a music-trained open-source audio model here:** Models such as CLAP, MuQ/MuLan, MERT, or Discogs EffNet are stronger choices when the product is indexing decoded audio waveforms or doing cross-modal text-to-audio retrieval. That is not the data shape in this assessment. The dataset contains remote MP3 URLs, not local audio files or precomputed audio features, so a proper audio-model implementation would need to download every file, decode audio, handle network and codec failures, add heavier ML dependencies, and usually accept higher CPU/GPU cost. It would also move effort away from the stated search-integrity problem: exact metadata search, sparse DynamoDB parsing, hybrid fusion, ranking, feedback concurrency, and lineage diversity. I would add a music-trained embedding model as a later sidecar once audio access and offline evaluation labels exist; the current schema supports that by versioning/replacing vectors through pgvector and a dimension migration.

**Seed data assumption:** The supplied export has no `created_at`, `clicks`, or `impressions`, but Parts 3 and the verification requirement need those values. `seed.py` uses the real records from the dataset and assigns deterministic synthetic ranking metadata by stable generation UUIDs: `Snowy Holiday Nights` (`29d8590d-9c4d-4c60-ad7b-d8138ad333f5`) is the fresh 3-day/40-click pop case, `Weekend Vibes` (`14c8d840-ae07-4b92-be72-f24adf3e11bc`) is the 2-year/1000-click old-pop case, `Kathma6's Anthem` is the low-confidence `1/1` case, and `Midnight Breeze` is the cold-start `0/0` case. This keeps the data handling real and stable even if the dataset order changes.

**Indexes:** `embedding` has an HNSW pgvector cosine index. `search_vector` has a GIN index. `metadata` has a JSONB GIN index for ad-hoc future filters. `lineage_id`, `primary_genre`, `bpm`, `musical_key`, and `created_at` are btree indexed. I did not index lyrics because lyrics are long and can swamp the metadata-heavy search use case in this assessment.

**TSVECTOR weighting:** title and exact technical fields are weight `A`; prompt and acoustic description are `B`; variant sounds are `C`; tags are `D`. A title/key/BPM/vocal match should beat a loose tag match.

**Considered and rejected:** A single `songs` table with both conversion paths in columns would simplify ingestion but makes output-level clicks and diversity awkward. A fully normalized tag schema was also unnecessary here; JSONB plus promoted high-value fields is easier to maintain under sparse evolving metadata.

**Known limitations:** The largest limitation is that MiniLM embeds metadata text, not audio. That is acceptable for this assessment because the provided dataset is metadata-first, but production audio search should eventually add an offline audio-embedding job. The other major limitation is that engagement and age fields are synthetic because the export does not include real clicks, impressions, or creation timestamps.

## Part 2 - Broken Search Query

**Fusion bug:** The production query uses `COALESCE(v.rank, 100)` and `COALESCE(f.rank, 100)`. A row missing from one retrieval channel still receives a fake reciprocal-rank contribution. In practice, vector-only rows get a lexical bonus even when they did not match the user's exact words, so exact keyword matches can be buried by semantically nearby but lexically wrong songs.

**FTS bug:** The query calls `to_tsquery('english', $2)` on user input. Raw searches like `C major`, `128 BPM`, or `female vocal` are not tsquery syntax and either error, require unsafe preprocessing, or behave differently from normal user search. If production catches the error and falls back to vector retrieval, users experience exactly what was reported: vibe queries work, exact terms almost disappear.

**Corrected query shape:** `search.py` uses `websearch_to_tsquery` for user text and gives absent channels a zero contribution:

```sql
WITH parsed_query AS (
    SELECT CAST(:query_embedding AS vector) AS query_embedding,
           websearch_to_tsquery('english', :query_text) AS fts_query
),
vector_ranked AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> (SELECT query_embedding FROM parsed_query), id) AS vector_rank
    FROM songs
    WHERE embedding IS NOT NULL
    ORDER BY embedding <=> (SELECT query_embedding FROM parsed_query), id
    LIMIT :retrieval_limit
),
fts_ranked AS (
    SELECT s.id,
           ROW_NUMBER() OVER (ORDER BY ts_rank_cd(s.search_vector, q.fts_query, 32) DESC, s.id) AS fts_rank
    FROM songs s CROSS JOIN parsed_query q
    WHERE numnode(q.fts_query) > 0
      AND s.search_vector @@ q.fts_query
    ORDER BY ts_rank_cd(s.search_vector, q.fts_query, 32) DESC, s.id
    LIMIT :retrieval_limit
),
fused AS (
    SELECT COALESCE(v.id, f.id) AS id,
           COALESCE(1.0 / (60 + v.vector_rank), 0.0) +
           COALESCE(1.15 / (60 + f.fts_rank), 0.0) AS hybrid_score
    FROM vector_ranked v
    FULL OUTER JOIN fts_ranked f ON v.id = f.id
)
SELECT s.*, fused.hybrid_score
FROM fused JOIN songs s ON s.id = fused.id
ORDER BY fused.hybrid_score DESC
LIMIT :limit;
```

I used a slight FTS weight because exact user-provided terms are a strong precision signal. In a production system I would tune that weight with offline judgments and click data.

**Query embedding guardrail:** `search.py` removes exact technical tokens such as `bpm`, `major`, `minor`, `female`, `male`, `vocal`, and single-letter musical keys from the text sent to MiniLM. The original query is still sent unchanged to Postgres full-text search. This is intentional: FTS is better for exact metadata terms like `C major`, `128 BPM`, and `female vocal`, while MiniLM is better for semantic/vibe terms like `sad rainy piano`, `festive upbeat pop`, or `chill study beat`.

**Simple example:** For `C major female vocal upbeat holiday pop`, the vector embedding focuses on `upbeat holiday pop`, while FTS still matches the full query including `C major female vocal`. Without this guardrail, MiniLM may treat `major`, `female`, or `vocal` as ordinary English words and add noisy vector matches that are semantically related to voices but not actually in `C major`. This keeps the hybrid system cleaner: vector search handles vibe, and FTS protects exact musical metadata.

**Considered and rejected:** I rejected keeping `to_tsquery` with manual escaping because it is easy to break on normal user input. I also rejected relying on vector search alone for exact terms because keys, BPM, and vocal attributes are metadata precision signals, not semantic vibe signals.

**Known limitations:** The RRF weights, FTS boost, and technical-token guardrail are heuristic. They are defensible for this dataset and verified examples, but production should tune them with offline relevance judgments and real click logs.

## Part 3 - Re-Ranking

**Decision made:** Final score is `hybrid_score * recency_multiplier * engagement_multiplier`.

Recency is exponential half-life with a 180-day half-life and a floor, so old songs are not zeroed out. Engagement is a confidence-weighted Wilson lower bound blended with an 8% prior CTR, so `40/60` is trusted much more than `1/1`, and `0/0` gets a cold-start prior instead of zero.

Current demo outputs:

| Song | Age | Clicks | Impressions | Hybrid | Final |
|---|---:|---:|---:|---:|---:|
| A | 3 days | 40 | 60 | 0.72 | about 0.765 |
| B | 2 years | 1000 | 5000 | 0.80 | about 0.543 |
| C | 1 day | 1 | 1 | 0.75 | about 0.668 |
| D | 6 months | 0 | 0 | 0.68 | about 0.514 |

**Considered and rejected:** Raw CTR was rejected because `1/1` would look perfect. Raw click count was rejected because it lets old popular songs dominate queries where users want newer relevant content.

**Major tradeoff:** Final ranking can intentionally move a lower-hybrid-score song above a higher-hybrid-score song. In the verification output, the fresh 3-day song has lower hybrid relevance than the 2-year-old song, but it wins after re-ranking because its engagement rate is much stronger and its recency is much better. This is the point of Part 3: hybrid score answers "is it relevant?", while final score answers "is it relevant, fresh, and trusted by users?"

**Known limitations:** The half-life, prior CTR, and engagement ranges are hand-chosen constants. They behave correctly on the required edge cases, but production should tune them against real search impressions, clicks, and position bias.

## Part 4 - Concurrency

**Decision made:** Feedback requests insert immutable rows into `feedback_events`. Inserts are idempotent by event id, so client retries with the same id become no-ops instead of double-counting or failing the request. A background flusher claims unprocessed rows with `FOR UPDATE SKIP LOCKED`, aggregates by `output_id`, updates `songs` once per batch, and marks events processed in the same transaction.

**Why this beats the naive update:** The naive design locks the same `songs` row on every click or impression with `UPDATE songs SET clicks = clicks + 1`. For a popular song, thousands of events can queue behind one row lock. This implementation inserts events first and updates aggregate song counters later in batches.

**Simple example:** If one song receives 100 clicks and 300 impressions, the naive approach performs 400 direct updates to the same `songs` row. This approach performs 400 inserts into `feedback_events`, then one flush aggregates them into one song update: `clicks = clicks + 100`, `impressions = impressions + 300`.

**Maximum staleness:** Counts feeding the re-ranker are stale by at most the flush interval plus flush runtime. The implementation defaults to 5 seconds. That is acceptable for search ranking; it avoids hot-row contention while keeping feedback fresh enough for discovery.

**Restart mid-flush:** The buffer is durable in Postgres, not process memory. If the process restarts before the transaction commits, rows remain `processed_at IS NULL` and are retried. If it commits, counts and processed marks commit together.

**Next bottleneck at 5,000 rps:** After row-lock contention is removed, the next bottleneck is insert/WAL throughput on `feedback_events` and connection pool pressure. The next production step would be time partitioning plus batched inserts or a queue/log such as Kafka before the database aggregate worker.

**Considered and rejected:** In-memory counters with periodic `UPDATE songs SET clicks = clicks + n` are faster but lose data on process restart. Per-request `UPDATE songs` was rejected because it is the original lock-contention problem.

**Known limitations:** This keeps the buffer in Postgres, which is durable and simple, but at sustained high write volume the next major bottleneck is insert/WAL throughput. A production version would likely add table partitioning, retention, and possibly a queue/log before the database aggregate worker.

## Part 5 - Diversity

**Decision made:** `diversify_results()` greedily selects from the re-ranked list while applying a duplicate-lineage penalty. For the first five slots it forces at least four distinct lineages when that many exist. If there are fewer than four lineages available, it fills the list with the best possible mix.

**Time complexity:** The implementation is `O(n^2)` because it rescans the remaining candidates for each output position. For 1,000 retrieved items that is fine in Python. If the candidate set grew substantially, I would switch to per-lineage priority queues and a heap.

**Known limitation:** Diversity is a post-processing step, so it cannot recover lineages that were never retrieved. Retrieval should still over-fetch enough candidates, which `search.py` does with a configurable `retrieval_limit`.

**Considered and rejected:** I rejected removing all duplicate lineages outright because a second variant may still be useful when there are not enough distinct lineages. I also rejected doing diversity only in SQL because the final ordering depends on Python re-ranking and engagement signals.

## Production Readiness Notes

**Decision made:** Runtime settings live in `settings.py` and are driven by environment variables with local Docker defaults. This keeps the assessment easy to run while avoiding hard-coded operational values inside database, seed, feedback, and search modules.

**Repository format:** `README.md` is the operator runbook, `.env.example` documents deploy-time configuration, and `DECISIONS.md` contains the required reasoning, rejected alternatives, and known limitations.

**Verification coverage:** `scripts/verify.ps1` runs the Docker Postgres setup, Alembic migration, Python compile check, and `seed.py` verification output. The saved `verification_output.log` and `verification_screenshot.png` show the two required assessment queries.

**Considered and rejected:** I did not move the core logic behind framework-specific classes because the assessment asks for independently reviewable modules. Keeping the domain logic in `search.py`, `ranking.py`, and `feedback.py` makes it testable without running an HTTP server.
